# Release Engineering

**Purpose:** Standardised, automated tooling for building, verifying, and publishing Beta releases of Batoi Classroom Suite.

The release toolkit at `scripts/release/` provides five scripts covering the full release lifecycle — from pre-flight checks through artifact verification to release notes generation — each designed to fail safely, detect missing dependencies, and return meaningful exit codes.

## Script Reference

| Script | Purpose | Exit Codes |
|---|---|---|
| `check-release.sh` | Pre-flight readiness checks before starting a release | 0=ready, 1=missing dep, 5=check failed |
| `build-release.sh` | Build wheel + sdist packages with SHA256 checksums | 0=ok, 1=missing dep, 2=build failed, 4=cleanup failed |
| `verify-package.sh` | Verify package metadata, contents, install, entry points, deps | 0=ok, 1=missing dep, 3=verify failed |
| `generate-release-notes.sh` | Generate release notes from CHANGELOG + git history | 0=ok, 1=missing dep |
| `verify-install.sh` | Full install cycle in a fresh virtual environment | 0=ok, 1=missing dep, 3=verify failed |

## Release Workflow

### Standard release (recommended)

```bash
# 1. Ensure working tree is clean
git status

# 2. Run pre-flight checks
./scripts/release/check-release.sh

# 3. Update VERSION (e.g. 0.2.0)
echo "0.2.0" > VERSION

# 4. Update version in pyproject.toml and CHANGELOG.md
#    - Move [Unreleased] entries under a new [0.2.0] section
#    - Update version in cli/pyproject.toml

# 5. Commit version bump
git add VERSION cli/pyproject.toml CHANGELOG.md
git commit -m "chore(release): bump version to 0.2.0"

# 6. Tag the release
git tag -a "v0.2.0" -m "v0.2.0"

# 7. Build artifacts
./scripts/release/build-release.sh

# 8. Verify package
./scripts/release/verify-package.sh

# 9. Generate release notes
./scripts/release/generate-release-notes.sh

# 10. Verify fresh install
./scripts/release/verify-install.sh --wheel dist/bcs-0.2.0-py3-none-any.whl

# 11. Push tag
git push origin "v0.2.0"

# 12. Publish to PyPI (future — manual step for now)
# python3 -m twine upload dist/*
```

### Quick build (for testing)

```bash
./scripts/release/build-release.sh --output /tmp/test-dist --skip-clean
./scripts/release/verify-package.sh --artifacts /tmp/test-dist --skip-install
```

## Artifact Verification

Every release build produces:

```
dist/
├── bcs-<version>-py3-none-any.whl       # Wheel package
├── bcs-<version>.tar.gz                  # Source distribution
└── SHA256SUMS.txt                        # Checksums for both
```

The `verify-package.sh` script verifies:

1. **Package metadata** — name, version, entry points match expectations
2. **Wheel contents** — all expected files present, no unexpected files
3. **Source distribution** — directory structure intact
4. **Editable install** — installs from source without errors
5. **Wheel install** — installs in a fresh virtual environment
6. **Entry points** — `bcs` command on PATH, `--version` and `--help` work
7. **Configuration files** — `config/schema.yaml` and `config/examples/default.yaml` present
8. **Dependency resolution** — `pip check` passes
9. **Dependency summary** — written to `reports/release/dependency-summary.txt`
10. **Installed file list** — written to `reports/release/installed-files.txt`

## Release Notes

Generated by `generate-release-notes.sh`:

```
reports/release/release-notes.md
```

Contains:

- Version and release date
- Changes since last tag (from CHANGELOG [Unreleased] section)
- Git commit summary grouped by type (features, fixes, docs, refactoring, etc.)
- Contributor list since last tag
- Artifact checksum table (from `dist/SHA256SUMS.txt`)
- Upgrade notes and installation instructions

## Beta Publication Checklist

### Pre-release

- [ ] All CI checks pass (Ruff, mypy, pytest)
- [ ] `VERSION` file updated and matches `cli/pyproject.toml`
- [ ] `CHANGELOG.md` has complete `[Unreleased]` section, ready to move
- [ ] Git working tree is clean
- [ ] All git tags are pushed
- [ ] `./scripts/release/check-release.sh` passes
- [ ] No placeholder fixture files remain (zero-byte `.txt` in `cli/tests/fixtures/`)

### Build

- [ ] `./scripts/release/build-release.sh` succeeds
- [ ] `dist/SHA256SUMS.txt` contains entries for all artifacts
- [ ] Artifact sizes recorded

### Verification

- [ ] `./scripts/release/verify-package.sh` passes
- [ ] `reports/release/dependency-summary.txt` generated
- [ ] `reports/release/installed-files.txt` generated
- [ ] `./scripts/release/verify-install.sh --wheel dist/bcs-*.whl` passes
- [ ] `bcs --version` reports correct version
- [ ] `bcs --help` lists all subcommands
- [ ] `bcs version`, `bcs doctor`, `bcs inventory` execute without errors

### Release notes

- [ ] `./scripts/release/generate-release-notes.sh` succeeds
- [ ] `reports/release/release-notes.md` has meaningful content
- [ ] Release notes reviewed and accurate

### Publication

- [ ] Git tag created (`git tag -a "v<version>" -m "v<version>"`)
- [ ] Tag pushed to origin (`git push origin "v<version>"`)
- [ ] GitHub Release created with release notes and artifacts
- [ ] PyPI publication (future — pip install bcs)

## Rollback Procedure

If a release must be rolled back:

1. **Revert version bump:** `git revert <version-bump-commit>`
2. **Reset tag:** `git tag -d "v<bad_version>" && git push origin :refs/tags/"v<bad_version>"`
3. **Restore CHANGELOG:** move released entries back under `[Unreleased]` or add a revert entry
4. **Re-run checks:** `./scripts/release/check-release.sh`
5. **Remove artifacts:** `rm -rf dist/` and rebuild with corrected version

## Future Extension Points

1. **Automated PyPI publishing** — `twine upload` integration, API token management via environment variables
2. **GitHub Release API integration** — create releases and upload artifacts via `gh` CLI
3. **Signed artifacts** — GPG signatures on wheel/sdist files for supply-chain security
4. **Multi-platform wheel builds** — `cibuildwheel` for Linux, Windows, macOS native wheels
5. **Release branch management** — automated branch creation (`release/v0.2.x`) and hotfix cherry-pick workflow
6. **SBOM generation** — Software Bill of Materials using `pip-licenses` or `cyclonedx-bom`
7. **Version auto-increment** — smart version bump from commit history (conventional commits)
8. **Pre-release channels** — separate alpha/beta/rc channels with different publication workflows
9. **CI/CD integration** — automatically run release toolkit from GitHub Actions when a tag is pushed
