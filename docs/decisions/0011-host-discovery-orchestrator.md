# ADR-0011: Host Discovery Orchestrator — Coordinating Discovery Adapters into Host Inventory

**Status:** Accepted

## Context

Two Host Discovery adapters exist or are designed today, independently of each other: the EFI Adapter (accepted, implemented — [ADR-0010](0010-efi-adapter-read-only-scope.md)) and the Storage Adapter (accepted, models implemented; `docs/STORAGE_ADAPTER.md`). Both documents independently ask, and neither answers, the same question: what assembles Discovery adapter output into something `bcs doctor`/`bcs inventory` can actually consume? [docs/EFI_ADAPTER.md § Open Questions](../EFI_ADAPTER.md#open-questions) and [docs/STORAGE_ADAPTER.md § Relationship to Existing Inventory Collectors](../STORAGE_ADAPTER.md#relationship-to-existing-inventory-collectors) each raise it, without reference to the other.

[docs/PLATFORM_LAYER.md § Open Questions](../PLATFORM_LAYER.md#open-questions--explicitly-deferred) also explicitly deferred "migrating `bcs.inventory.collectors` to accept an injected `CommandRunner`" as its own follow-on decision, "once a first real adapter... is actually being built" — that condition is now met twice over.

Beyond the two adapters that exist today, this project anticipates several more Discovery domains (Secure Boot, Filesystem, Network, CPU, Memory, TPM — the latter three of which already have non-Discovery, `sysfs`-based collectors in `bcs.inventory.collectors`). Without a coordinating component, each future adapter would independently re-ask the same "how does this reach `HostInventory`" question `EFI_ADAPTER.md` and `STORAGE_ADAPTER.md` already asked, and nothing would prevent `bcs.inventory.service.collect_host_inventory()` from accumulating an ad hoc, ever-growing sequence of direct adapter calls with no consistent error-isolation or extensibility story.

## Decision

1. **A new component, the Host Discovery Orchestrator (`bcs.inventory.discovery`), coordinates every Discovery adapter and aggregates their output into a new `HostDiscoverySnapshot` model.** Full design in [docs/HOST_DISCOVERY_ORCHESTRATOR.md](../HOST_DISCOVERY_ORCHESTRATOR.md).
2. **It depends only on already-bound, zero-argument adapter callables (`HostDiscoveryAdapters`), never on `CommandRunner` or `subprocess` directly.** Binding a tool-based adapter's `read_*` function to a live `CommandRunner` happens once, at `bcs.app.main()`'s composition root — the same place `SubprocessCommandRunner` itself is already built.
3. **Eight explicit, named, optional slots — not a dynamic registry.** EFI, Storage, Secure Boot, Filesystem, Network, CPU, Memory, TPM. An unset slot (`None`) means "no adapter wired for this domain in this build," not an error. Adding a ninth domain later is a small, reviewed, additive change to two data structures, not a runtime extension point — the same reasoning [docs/HOST_INVENTORY.md § Proposed Changes, item 4](../HOST_INVENTORY.md#proposed-changes-requiring-approval) already used to decline a dynamic `Collector` protocol/registry.
4. **The orchestrator never interprets, ranks, selects, or merges adapter output.** It has no concept of a preferred disk, ESP, boot order, or installation target — every `HostDiscoverySnapshot` field is either exactly what an adapter returned or absent. Any future decision of that kind is a separate, later component consuming this orchestrator's output, not this one.
5. **Per-domain failure isolation.** A wired adapter raising a `bcs.platform.errors.PlatformError` (or subclass) leaves that one domain's field absent and records one `caveats` entry; it never prevents the other domains from being collected. Any other exception propagates uncaught — this component does not widen what counts as an "expected" failure beyond the Platform Layer's own typed hierarchy.
6. **This decision does not itself add new fields to `HostInventory`.** `HostDiscoverySnapshot` is produced first; folding its new-style facts (`firmwareBootConfiguration`, `storageTopology`, ...) into `HostInventory` as additive fields is a separate, later, explicitly flagged follow-up — mirroring how [ADR-0008](0008-host-inventory-ports-and-adapters.md)'s own EFI System Partition/USB Storage amendment was recorded and approved independently of that ADR's original acceptance.
7. **`bcs.inventory.service.collect_host_inventory()` gains an `orchestrator: HostDiscoveryOrchestrator` parameter** — a real, visible signature change to existing, tested code, explicitly acknowledged here rather than left implicit.

### Alternatives Considered

- **A dynamic `Collector`-style adapter registry**, allowing adapters to self-register rather than being named as explicit dataclass fields. Rejected for the same reason [docs/HOST_INVENTORY.md § Proposed Changes, item 4](../HOST_INVENTORY.md#proposed-changes-requiring-approval) already rejected it for collectors: no concrete second contributor motivates it, and it is exactly the kind of speculative flexibility [REVIEW.md §7](../../REVIEW.md#7-a-meta-concern-proportionality) argues against building ahead of a real need. All eight domains this orchestrator coordinates are already known and named by this decision itself.
- **Have the orchestrator return `HostInventory` directly**, instead of a dedicated `HostDiscoverySnapshot`. Rejected: `HostInventory` also carries `identity` and `tooling`, neither of which is among the eight Discovery domains in this orchestrator's scope and neither of which has (or needs) a Discovery adapter. Forcing the orchestrator to also own those two fields would be scope creep beyond "coordinate discovery adapters." A dedicated output type keeps the orchestrator's own responsibility exactly as narrow as its name states.
- **Fold orchestration directly into `bcs.inventory.service.collect_host_inventory()`** (a bigger function, no separate class). Rejected: the DI/testability benefit of a constructor-injected `HostDiscoveryAdapters` bundle — tests substitute stub callables without patching module state — is the same benefit already established for `CommandRunner` itself ([ADR-0009](0009-platform-layer-command-runner.md)); collapsing it back into a plain function would lose that seam for no offsetting benefit.
- **Let `bcs doctor` also depend on the full `HostDiscoveryOrchestrator.discover()` sweep**, for consistency with `bcs inventory`. Rejected: `doctor`'s existing, deliberate asymmetry (evaluating one fact at a time, independent of the other checks, per [docs/HOST_INVENTORY.md § Dependency Graph](../HOST_INVENTORY.md#dependency-graph)) would be lost if a single `--check` had to pay for, or could be blocked by, an unrelated domain's adapter call. `doctor` instead reads one named slot off `HostDiscoveryAdapters` directly, preserving that asymmetry.
- **Parallelize the up-to-eight adapter calls within one `discover()` sweep.** Considered, not designed: with two adapters implemented/designed today, there is no concrete performance problem to solve yet, and introducing concurrency (thread pool, cross-thread error aggregation) ahead of that need is the same proportionality risk this ADR otherwise argues against elsewhere.

## Consequences

- Both `docs/EFI_ADAPTER.md § Open Questions` and `docs/STORAGE_ADAPTER.md § Relationship to Existing Inventory Collectors` gain a concrete answer to their shared, previously-independent open question — though neither document is amended by this ADR to say so; that cross-reference lives in [docs/HOST_DISCOVERY_ORCHESTRATOR.md § Related Documents](../HOST_DISCOVERY_ORCHESTRATOR.md#related-documents) instead.
- `bcs.inventory.collectors.collect_cpu`/`collect_memory`/`collect_network` need **no signature change** to participate — they are already zero-argument functions and satisfy `HostDiscoveryAdapters`' callable shape as-is. Only tool-based adapters (`efi`, and later `storage`, `secure_boot`, `filesystem`, `tpm`) need explicit binding to a `CommandRunner` at the composition root.
- `RuntimeContext` is expected to gain a `host_discovery_orchestrator` field, the same treatment `command_runner` received in Platform-001 Part 4 — a real, additive change to an already-implemented, frozen dataclass, not yet made.
- `HostInventory`'s own schema is **not** changed by this ADR. A follow-up proposal (amending [docs/HOST_INVENTORY.md](../HOST_INVENTORY.md), likely as an ADR-0008 amendment, mirroring its EFI System Partition/USB Storage precedent) is required before `bcs inventory`'s JSON output reflects any Discovery adapter's data. Accepting this ADR does not itself approve that follow-up.
- This ADR's acceptance approves the architecture described in [docs/HOST_DISCOVERY_ORCHESTRATOR.md](../HOST_DISCOVERY_ORCHESTRATOR.md). **Implemented so far (Part 1 — data-holding types only):** `HostDiscoveryAdapters` and `HostDiscoverySnapshot` (`cli/src/bcs/inventory/discovery/models.py`). **Not yet implemented:** `HostDiscoveryOrchestrator` itself (`orchestrator.py`), the composition-root wiring that would bind real adapter callables into `HostDiscoveryAdapters`, `RuntimeContext` integration, and `bcs.inventory`/`bcs inventory` integration — each remains gated on its own separate, explicit go-ahead, per this project's established pattern for adapter designs ([ADR-0009](0009-platform-layer-command-runner.md), [ADR-0010](0010-efi-adapter-read-only-scope.md)).
- **No effect on already-accepted architecture.** `bcs.platform.models.CommandResult`, the `PlatformError` hierarchy, `bcs.platform.execution.CommandRunner`/`SubprocessCommandRunner`, `bcs.context.RuntimeContext.command_runner`, `bcs.inventory.models.HostInventory`'s existing fields, and both adapters' own already-accepted designs (`FirmwareBootConfiguration`/`BootEntry`, and the Storage Adapter's implemented models) are unchanged — this ADR only adds a new, additive coordination layer beneath them.
