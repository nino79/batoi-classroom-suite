# ADR-0010: EFI Adapter â€” Read-Only, Domain-Named Firmware Boot Configuration Integration

**Status:** Accepted

## Context

Host Inventory ([ADR-0008](0008-host-inventory-ports-and-adapters.md)) can report *whether* a machine is UEFI-capable and its Secure Boot state, but has no visibility into actual UEFI boot entries (`BootOrder`, `BootCurrent`, individual `Boot####` entries). Boot Manager's own architecture ([docs/architecture/boot-manager.md Â§ Open Questions](../architecture/boot-manager.md#open-questions)) already flags UEFI NVRAM boot-entry management (`BM-003`) as unresolved, with reliability known to vary significantly by OEM firmware.

The Platform Layer ([ADR-0009](0009-platform-layer-command-runner.md)) already anticipated this exact adapter in [docs/PLATFORM_LAYER.md Â§ How Future Adapters Use It](../PLATFORM_LAYER.md#how-future-adapters-use-it), naming `efibootmgr` as the first candidate. This ADR is the first adapter built on that foundation â€” the start of a broader **Host Discovery** effort (read-only inspection adapters), of which this is the first piece.

Building a read-only inspection capability first, before any UEFI boot-entry *write* capability is designed, lets BCS observe real boot-entry data across real classroom hardware before committing to how (or whether) BCS should ever change it â€” a deliberately cautious sequencing given how firmware-quirky NVRAM boot-entry management can be.

This ADR was reviewed and accepted with six amendments, incorporated directly into the Decision below rather than recorded as a later addendum, since nothing was implemented between proposal and acceptance (contrast [ADR-0008](0008-host-inventory-ports-and-adapters.md), whose amendment came after code already existed).

## Decision

1. **This adapter is permanently read-only.** It never modifies EFI variables, never changes boot order, and never installs rEFInd or any other boot loader/manager. The only `efibootmgr` flag it ever passes is `-v`. Full design in [docs/EFI_ADAPTER.md](../EFI_ADAPTER.md).
2. **The package is domain-named: `bcs.platform.adapters.efi`, not `.efibootmgr`.** It represents the EFI domain, not the current implementation technology; a future backend (`efivarfs`, `libefivar`, or anything else) could replace `efibootmgr` internally without a public rename. This is the first application of a new, project-wide rule â€” see point 4.
3. **A pure-function parser, completely independent of the execution layer.** `parse_firmware_boot_configuration(text: str) -> FirmwareBootConfiguration` accepts only a string, produces only immutable Pydantic models, never imports `CommandRunner` or `subprocess`, and never knows where its input text came from â€” it is designed to be extractable as a standalone library with no changes. `read_firmware_boot_configuration(runner: CommandRunner) -> FirmwareBootConfiguration` is a thin orchestration layer that calls it. This maximizes what's testable without mocking (see [docs/EFI_ADAPTER.md Â§ Testing Strategy](../EFI_ADAPTER.md#testing-strategy)).
4. **A new project-wide architectural rule: domain-driven naming.** *"Packages, adapters, and Pydantic models shall be named after the business/domain concept, never after the implementation technology."* Recorded in [docs/standards/naming-conventions.md Â§ Domain-Driven Naming](../standards/naming-conventions.md#domain-driven-naming) and [docs/standards/coding-standards.md Â§ Domain-Driven Naming](../standards/coding-standards.md#domain-driven-naming), and cross-referenced from [ARCHITECTURE.md Â§5](../../ARCHITECTURE.md#5-cross-cutting-concerns). Applied here: `EfiBootConfiguration` â†’ `FirmwareBootConfiguration` (see [docs/EFI_ADAPTER.md Â§ Pydantic Models](../EFI_ADAPTER.md#pydantic-models) for the specific naming-collision rationale), `EfibootmgrError`/`EfibootmgrUnavailableError`/`EfibootmgrParseError` â†’ `FirmwareBootError`/`FirmwareBootUnavailableError`/`FirmwareBootParseError`. `BootEntry` required no change â€” it was already domain-named.
5. **The locale policy is promoted from an EFI-specific detail to a Platform Layer-wide rule**: every adapter executing a Linux command via `CommandRunner` MUST force `LANG=C`/`LC_ALL=C`, building a full `os.environ` copy plus the two overrides (`CommandRunner`'s `env` replaces rather than merges â€” [docs/PLATFORM_LAYER.md Â§ CommandRunner API](../PLATFORM_LAYER.md#commandrunner-api)). Recorded in [docs/PLATFORM_LAYER.md Â§ Locale Policy](../PLATFORM_LAYER.md#locale-policy); this adapter is its first conforming example, not its origin.
6. **Command execution metadata does not belong on `FirmwareBootConfiguration`.** Evaluated and rejected on domain-purity grounds (the model represents a fact about the machine, not about the act of probing it) and because the pure parser that builds it structurally has no access to execution facts. A reusable `CommandMetadata`/envelope concept for future adapters is documented as a deferred idea in [docs/EFI_ADAPTER.md Â§ Command Metadata](../EFI_ADAPTER.md#command-metadata), not concluded to be architecturally necessary yet, and not designed further here.
7. **Internally organized as a small subpackage** (`bcs/platform/adapters/efi/{models,parser,adapter,errors}.py`) rather than the single flat file [docs/PLATFORM_LAYER.md Â§ Package Structure](../PLATFORM_LAYER.md#package-structure) originally sketched for adapters in general. The public import surface (`bcs.platform.adapters.efi`) is unchanged either way.

### Alternatives Considered

- **Keep the package named `efibootmgr`**, since that is the concrete tool it wraps today. Rejected: it ties a public, durable name to a specific, swappable implementation choice, and would force a breaking rename if the backend ever changes â€” exactly the coupling [ADR-0009](0009-platform-layer-command-runner.md)'s ports-and-adapters discipline already argues against one layer up.
- **Name the top-level model `BootConfiguration`** (the shortest, most generic option). Rejected: it collides with the already-existing, unrelated concept of Boot Manager's own menu configuration (`spec.bootManager.menu` in `ClassroomConfig`, see [docs/CONFIGURATION.md](../CONFIGURATION.md)) â€” a real ambiguity risk, not a hypothetical one. `FirmwareBootConfiguration` is unambiguous against that term.
- **Name entries `BootOption`** (the UEFI specification's own term). Considered equally valid; not chosen because "boot option" is common shorthand for an unrelated concept (a kernel/bootloader command-line option, as in GRUB) that this model has nothing to do with. `BootEntry` avoids that collision.
- **Fail (raise) on any unrecognized parser line**, rather than permissively skipping it. Rejected: `efibootmgr`'s output format has drifted across versions before and will again; a strict parser would make this adapter brittle against exactly the kind of minor version drift [docs/EFI_ADAPTER.md Â§ Supported `efibootmgr` Versions](../EFI_ADAPTER.md#supported-efibootmgr-versions) already expects to tolerate on a best-effort basis.
- **Merge `LC_ALL`/`LANG` on top of a minimal environment** instead of a full `os.environ` copy. Rejected: `CommandRunner`'s `env` replaces entirely; a minimal environment would drop `PATH` and anything else the child process needs, breaking execution rather than just locale.
- **Wrap `CommandNotFoundError`/`CommandTimeoutError` in adapter-specific equivalents**, for a uniform "every error is `FirmwareBoot*Error`" surface. Rejected as needless indirection: both already are clean, typed, Platform Layer exceptions; wrapping them would add a layer with no new information and would require callers to unwrap to get at the same data `CommandRunner` already exposes.
- **Add execution-metadata fields directly to `FirmwareBootConfiguration`** (duration, timestamps, exit code). Rejected â€” see Decision point 6; the model represents a machine fact, the pure parser that builds it cannot supply execution facts even if the schema had room for them, and `CommandResult` already carries this data without duplication.
- **Design UEFI boot-entry write/management capability alongside this read-only adapter**, since they're topically related. Rejected: read and write are genuinely different risk profiles (nothing here can damage a machine's boot configuration; a write-capable tool could). Bundling them would also make this ADR's read-only guarantee harder to audit at a glance. A write capability, if ever pursued, gets its own document and its own ADR.
- **A flat `adapters/efi.py` file**, matching `docs/PLATFORM_LAYER.md`'s original sketch literally. Considered acceptable but not chosen: this adapter's internal structure (schema, pure parser, I/O adapter, errors) is developed enough to benefit from separation, and the public import surface is unaffected either way.

## Consequences

- `bcs doctor`/`bcs inventory` gain a path to real UEFI boot-entry visibility, closing a documented gap â€” but only once a *separate*, not-yet-made decision wires this adapter into either of them (see [docs/EFI_ADAPTER.md Â§ Open Questions](../EFI_ADAPTER.md#open-questions)); accepting this ADR does not itself decide that.
- `FirmwareBootConfiguration`/`BootEntry` join `CommandResult` as models with no `schemaVersion` of their own, since neither is ever expected to be a `bcs` command's own top-level payload â€” consistent with the precedent [ADR-0009](0009-platform-layer-command-runner.md) already established.
- `bcs.platform.errors.PlatformError` gains its first adapter-level subclasses (`FirmwareBootError` and its two children); any future adapter following this same shape is expected to do the same, keeping a single `except PlatformError` catchable across the whole Platform Layer.
- **The domain-driven-naming rule (point 4) is now project-wide, not EFI-specific.** Future adapters (`lsblk`, `blkid`, `mount`, `rsync` â€” still placeholder tool-names in `docs/PLATFORM_LAYER.md`, not yet individually designed) are expected to receive domain-appropriate names of their own when designed, not to keep their current tool-derived placeholder names by default.
- **The locale policy (point 5) is now project-wide, not EFI-specific.** Every future Platform Layer adapter that executes a Linux command is expected to force `LANG=C`/`LC_ALL=C` the same way, per [docs/PLATFORM_LAYER.md Â§ Locale Policy](../PLATFORM_LAYER.md#locale-policy).
- This ADR's acceptance approves the architecture described in [docs/EFI_ADAPTER.md](../EFI_ADAPTER.md), including its Package Structure/Pydantic Models/Parser Architecture/Adapter Responsibilities/Error Mapping/Testing Strategy sections. **None of it is implemented.** Implementation remains gated on a separate, explicit go-ahead before any `.py` file is touched.
- **No effect on already-implemented Platform-001 code.** `bcs.platform.models.CommandResult`, `bcs.platform.errors.PlatformError`/`CommandNotFoundError`/`CommandTimeoutError`/`CommandExecutionError`, `bcs.platform.execution.CommandRunner`/`SubprocessCommandRunner`, and `bcs.context.RuntimeContext.command_runner` are all unchanged â€” this ADR only adds new, adapter-scoped names underneath them. See [docs/EFI_ADAPTER.md Â§ Backward Compatibility with Platform-001](../EFI_ADAPTER.md#backward-compatibility-with-platform-001) for the explicit, itemized check.
- The read-only guarantee (Â§ Decision, point 1) remains a hard constraint on any future change to this adapter, not just its initial version â€” a later contributor proposing to add a write flag to this same module would be reversing this ADR, not extending it, and would need its own ADR to do so.
