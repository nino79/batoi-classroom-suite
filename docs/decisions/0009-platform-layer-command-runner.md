# ADR-0009: Platform Layer as the Sole Path to Process Execution

**Status:** Accepted

## Context

BCS's Python code (`cli/`) already has two shapes of "talk to the operating system": `bcs.plugins.run_plugin` calls `subprocess.run` directly to dispatch to external `bcs-<name>` plugins, and `bcs.inventory.collectors` deliberately avoids external tools entirely, reading `/proc`/`/sys` files instead — with documented placeholder gaps (IP address enumeration, Secure Boot byte-value parsing) that a real tool (`ip`, an EFI-variable-aware utility) would resolve better than a file read can. Future work will need to shell out to Linux tools with no `/proc`/`/sys` equivalent at all: `efibootmgr` (UEFI boot entries, `BM-003`), `lsblk`/`blkid` (richer storage/partition detail than today's collectors), `mount`/`umount` (Deploy's disk-layout restoration, `DEP-003`), `rsync` (artifact staging).

Without a single, deliberate seam, each of these would reinvent command construction, timeout handling, output capture, and error translation independently — the same duplicated-concept risk [REVIEW.md](../../REVIEW.md) already exists to catch, and the same reasoning that motivated [ADR-0008](0008-host-inventory-ports-and-adapters.md) for Host Inventory (one probe per fact, in one place) applied one layer lower (one execution path, in one place).

This ADR was reviewed and accepted with five amendments, incorporated directly below rather than recorded as a later addendum, since nothing was implemented between proposal and acceptance (contrast [ADR-0008](0008-host-inventory-ports-and-adapters.md), whose amendment came after code already existed).

## Decision

1. **All process execution from BCS's Python code goes through a Platform Layer (`bcs.platform`), never `subprocess` directly.** `bcs.platform.execution` is the only module permitted to import `subprocess`; every collector, command, and future adapter depends on `bcs.platform.CommandRunner` instead. Full design in [docs/PLATFORM_LAYER.md](../PLATFORM_LAYER.md). *(The implementation module is named `execution.py`, not `runner.py` — renamed during review; `CommandRunner`/`SubprocessCommandRunner` class names are unaffected.)*
2. **`CommandRunner` is a structural interface (`Protocol`) with two implementations from day one**: `SubprocessCommandRunner` (production) and `FakeCommandRunner` (test-only). Consumed via dependency injection (a parameter, added to `RuntimeContext`), never a module-level singleton or monkeypatched module state.
3. **`CommandResult` carries `command`, `stdout`, `stderr`, `exitCode`, `duration`, `startedAt`, `finishedAt`, `workingDirectory`, and `timedOut`, plus a computed `success` property** (`exitCode == 0 and not timedOut`) — extended during review from an initial narrower field set. Full field-by-field detail in [docs/PLATFORM_LAYER.md § Models](../PLATFORM_LAYER.md#models).
4. **Commands are always represented as an argument list; `shell=True` is never used, anywhere, under any circumstance.** This is a named architectural rule, not an implicit consequence of "structured in, structured out" — see [docs/PLATFORM_LAYER.md § Architectural Rule: Argument Lists Only](../PLATFORM_LAYER.md#architectural-rule-argument-lists-only-never-shelltrue). It eliminates shell-injection risk (Bandit `S602`) by construction: there is no code path that accepts or interpolates a shell string, in `bcs.platform.execution` or in any adapter built on it.
5. **The Platform Layer is independent of `bcs.errors`/`ExitCode`.** It defines its own exception hierarchy (`PlatformError` and subclasses); translating a Platform Layer exception into a specific CLI exit code is each calling command's own responsibility, exactly the same core/adapter independence [ADR-0008](0008-host-inventory-ports-and-adapters.md) established for Host Inventory.
6. **A new non-functional requirement, `NFR-008`**, is added to [SPECIFICATION.md §3](../../SPECIFICATION.md#3-non-functional-requirements), reusing the existing `NFR-` prefix rather than introducing a new one: *"Every external command invocation from BCS's Python code MUST be executed through the Platform Layer's `CommandRunner` — centrally logged and timeout-bounded — never invoked ad hoc via `subprocess` from business/command code."* No new requirement-ID prefix is introduced.
7. **`docs/standards/coding-standards.md` states the prohibition explicitly and by tool name**: direct use of `subprocess.run()`, `subprocess.Popen()`, or equivalent (`os.system`, `os.popen`, `os.exec*`) outside `bcs.platform` is forbidden, not merely discouraged.
8. **`cli/pyproject.toml`'s `S603`/`S607` Ruff ignores narrow from repository-wide to exactly two files**: `bcs.plugins` (see point 9) and `bcs.platform.execution`. Any other direct `subprocess` usage fails CI lint automatically — a mechanical guardrail, not just a reviewed convention.
9. **`bcs.plugins.run_plugin`'s existing direct `subprocess.run` call remains a deliberate, reviewed exception, not a bug to fix.** Plugin dispatch needs full passthrough stdio (an interactive plugin must be able to prompt the user directly); `CommandRunner` is built around captured output for structured parsing, a different contract.

### Alternatives Considered

- **No central abstraction; review each `subprocess` call case by case**, as today's blanket `S603`/`S607` ignore effectively does. Rejected: this is exactly the duplicated-effort risk motivating this ADR, and provides no mechanical enforcement — a future contributor could add a new direct `subprocess` call anywhere and nothing would catch it.
- **An abstract base class (`abc.ABC`) instead of a `Protocol`.** Rejected as unnecessary ceremony for a structural-typing use case with no shared implementation to inherit — a `Protocol` expresses "anything with this shape" without forcing `FakeCommandRunner` into an inheritance relationship it doesn't need.
- **Route plugin dispatch through `CommandRunner` too, for uniformity.** Rejected: it would require either weakening `CommandRunner`'s captured-output contract or adding a passthrough mode with no other consumer, purely for the sake of "everything goes through one thing" — the kind of premature generalization [REVIEW.md §7](../../REVIEW.md#7-a-meta-concern-proportionality) already argues against for this project.
- **A new requirement-ID prefix (e.g., `SYS-`) for Platform Layer requirements.** Rejected: `PLAT-` is already taken (target platform/hardware matrix, [SPECIFICATION.md §1](../../SPECIFICATION.md#1-target-platform-matrix)) and would collide in spirit if not in literal ID; the Platform Layer's one externally-observable property (centralized, audited, timeout-bounded execution) fits the existing `NFR-` prefix without strain.
- **Naming the module `runner.py`** (the original proposal). Renamed to `execution.py` during review — no alternative rationale recorded beyond the maintainers' explicit naming preference; `CommandRunner` as the interface/class name is retained.
- **Deriving `duration` from `started_at`/`finished_at` instead of storing it explicitly.** Considered during the `CommandResult` field-extension amendment; kept as an explicit field per the maintainers' direction (`Include: ... duration ...`) despite the redundancy, so a consumer doesn't have to compute it — flagged transparently in [docs/PLATFORM_LAYER.md § Models](../PLATFORM_LAYER.md#models) rather than silently resolved.

## Consequences

- Every future adapter (`efibootmgr`, `lsblk`, `blkid`, `mount`, `rsync`) is testable against a `FakeCommandRunner` without touching a real OS/device, the same benefit `bcs.inventory.collectors`' `tmp_path`-backed filesystem faking already provides for file-based facts.
- `bcs.inventory.collectors`' two documented placeholder gaps (IP enumeration, Secure Boot byte-value parsing) become closeable via a `lsblk`/`ip`-based or EFI-variable-aware adapter — but migrating existing collector signatures to accept an injected `CommandRunner` is a real, visible change to already-implemented, already-tested code, and is explicitly deferred as its own follow-on decision (see [docs/PLATFORM_LAYER.md § Open Questions](../PLATFORM_LAYER.md#open-questions--explicitly-deferred)), not implied by accepting this ADR.
- `cli/pyproject.toml`'s Bandit `S603`/`S607` ignores become narrower and therefore more meaningful; any new direct `subprocess` usage outside the two named exceptions fails CI lint automatically, once implemented.
- `CommandResult`'s richer field set (timestamps plus duration) makes it a better fit for a future Deploy session report (`DEP-005`) than the original, narrower proposal — a session report can now show exactly when each command ran, not just how long it took.
- This ADR's acceptance approves the architecture described in [docs/PLATFORM_LAYER.md](../PLATFORM_LAYER.md), including its "Approved Design Decisions (Pending Implementation)" list — unlike [ADR-0008](0008-host-inventory-ports-and-adapters.md)'s acceptance, which left Host Inventory's proposed-changes list separately gated. **None of it is implemented.** Implementation remains gated on a separate, explicit go-ahead before any `.py` or `pyproject.toml` file is touched.
