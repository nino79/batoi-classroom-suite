# Implementation Patterns for Platform Layer Adapters

## 1. Purpose

The EFI, Storage, and Secure Boot adapters — the three Host Discovery adapters implemented so far (see [IMPLEMENTATION_STATUS.md § 4. Platform Adapter Matrix](IMPLEMENTATION_STATUS.md#4-platform-adapter-matrix)) — were each built independently, over several sessions, by different reasoning about the same underlying design. The result converged on a single, repeatable workflow, but that workflow has only ever existed *implicitly*: scattered across three design documents, four ADRs, and dozens of `CHANGELOG.md` entries, each rediscovering (or re-explaining) the same shape.

This document makes that workflow explicit, so the next adapter — Filesystem, Network, CPU, Memory, TPM, or any adapter not yet named — is built by following a written process rather than by re-reading three prior implementations and inferring the pattern again.

**Relationship with [AGENTS.md](../AGENTS.md):** AGENTS.md is the authoritative, project-wide contract — scope, hard constraints, and workflow rules that apply to *everything* in this repository, not just Platform Layer adapters. This document does not restate or override any of it; where the two could be read as being in tension, AGENTS.md wins, per its own [§ Tool Precedence](../AGENTS.md#tool-precedence) principle applied one level down. What this document adds is the adapter-specific *sequencing and detail* AGENTS.md deliberately leaves general — e.g. AGENTS.md's [§ Workflow](../AGENTS.md#workflow) says "design first, implementation later" and requires a design document before code; this document says exactly what sections that design document needs and what "later" is actually made of, stage by stage.

**Relationship with [PLATFORM_LAYER.md](PLATFORM_LAYER.md):** PLATFORM_LAYER.md is the architecture — the `CommandRunner`/`CommandResult`/`PlatformError` core every adapter is built on, and the [Locale Policy](PLATFORM_LAYER.md#locale-policy), [Argument Lists Only](PLATFORM_LAYER.md#architectural-rule-argument-lists-only-never-shelltrue), and [How Future Adapters Use It](PLATFORM_LAYER.md#how-future-adapters-use-it) rules every adapter must satisfy. This document does not define or alter any of that architecture — it is the *process* document for building something that satisfies it: the order of work, the checklist, the test philosophy, and the mistakes already made once and worth not repeating.

This document only covers Platform Layer adapters. It says nothing about Boot Manager, Builder, or Deploy — those remain Bash, per [ADR-0004](decisions/0004-bash-as-primary-implementation-language.md), and out of scope for `cli/` entirely per [AGENTS.md Hard Constraint 1](../AGENTS.md#hard-constraints).

## 2. Adapter Lifecycle

The complete lifecycle, idea to finished implementation. Every stage below is required; skipping one is exactly the kind of gap [§ 7. Common Mistakes](#7-common-mistakes) documents.

1. **Design proposal.** A new `docs/<DOMAIN>_ADAPTER.md`, matching the structure in [§ 5. Documentation Updates](#5-documentation-updates), written *before* any code — per [AGENTS.md § Workflow](../AGENTS.md#workflow): "a new Platform Layer component... gets a design document under `docs/`... before any code." Status banner starts `Proposed, pending approval`. See [docs/FILESYSTEM_ADAPTER.md](FILESYSTEM_ADAPTER.md) for the current example of this stage, not yet past it.
2. **ADR evaluation.** Decide — and record the reasoning in the design document's own "ADR Recommendation" section — whether this adapter needs a new ADR, per [AGENTS.md § ADR Workflow](../AGENTS.md#adr-workflow) and [docs/decisions/README.md § When to Write an ADR](decisions/README.md#when-to-write-an-adr). In practice, every adapter built so far has concluded **no** new ADR is required, because every architectural mechanism it uses (read-only scope, domain-driven naming, the pure-parser/thin-adapter split, the `PlatformError`-rooted exception hierarchy, the Platform Layer locale policy) was already decided by [ADR-0008](decisions/0008-host-inventory-ports-and-adapters.md)/[ADR-0009](decisions/0009-platform-layer-command-runner.md)/[ADR-0010](decisions/0010-efi-adapter-read-only-scope.md) — see [EFI_ADAPTER.md](EFI_ADAPTER.md), the ADR-0010 result itself; [STORAGE_ADAPTER.md § ADR Recommendation](STORAGE_ADAPTER.md#adr-recommendation); [SECURE_BOOT_ADAPTER.md § ADR Recommendation](SECURE_BOOT_ADAPTER.md#adr-recommendation). A new ADR is genuinely needed only when a design introduces a naming collision risk significant enough to want its own durable record, or changes a component boundary — not merely "applies an existing pattern to a new domain."
3. **Design review.** An independent read of the design document against the accepted architecture and its siblings, checking specifically: discovery-vs-decision boundaries, dependency direction, naming consistency, domain-model purity, parser purity, adapter responsibilities, future extensibility, and duplication with existing adapters. Review-only — no code, no document edits during this stage. This is where the Filesystem Adapter's own partial-failure design was caught and reworked before implementation began (see its `raw_stderr` field and [§ Relationship to the Host Discovery Orchestrator's Caveats Model](FILESYSTEM_ADAPTER.md#relationship-to-the-host-discovery-orchestrators-caveats-model)).
4. **Models.** `models.py` — the domain's Pydantic model(s), frozen, `extra="forbid"`, `populate_by_name=True`, `camelCase` JSON aliases. No parsing logic, no `subprocess`, no `CommandRunner` import, nothing that executes anything. See [§ 6. Architecture Rules](#6-architecture-rules).
5. **Parser.** `parser.py` — a pure function, `text: str` in, an immutable model out, permissive by default (see [§ 4. Testing Strategy](#4-testing-strategy)). Never imports `errors.py`, `CommandRunner`, or `subprocess`.
6. **Errors.** `errors.py` — the domain's exception hierarchy, rooted at `bcs.platform.errors.PlatformError`, following the `<Domain>Error` / `<Domain>UnavailableError` / `<Domain>ParseError` shape every adapter so far has used.
7. **Adapter.** `adapter.py` — the one module that imports `CommandRunner` and actually executes a process; translates raw exit codes/`stderr` into the typed exceptions from stage 6, and hands successful output to the parser from stage 5.
8. **Package exports.** `__init__.py` re-exports every public name from stages 4–7 (models, errors, the parser function, the adapter function) — see [§ 7. Common Mistakes](#7-common-mistakes) for what happens when this is skipped.
9. **Composition Root wiring.** Bind the new `read_*` function into `HostDiscoveryAdapters` in `bcs.app.main()` (`cli/src/bcs/app.py`), sharing the single `SubprocessCommandRunner` instance every other tool-based slot already shares, via `functools.partial(read_<domain>, runner=command_runner)`.
10. **Host Discovery integration.** Narrow the corresponding `HostDiscoveryAdapters`/`HostDiscoverySnapshot` slot in `cli/src/bcs/inventory/discovery/models.py` from the generic placeholder type (`Callable[[], object] | None` / `object | None`) to the adapter's own concrete model type. No change to `HostDiscoveryOrchestrator` itself is ever needed here — its coordination logic is domain-agnostic by design (see [HOST_DISCOVERY_ORCHESTRATOR.md § Aggregation-Only Guarantee](HOST_DISCOVERY_ORCHESTRATOR.md#aggregation-only-guarantee)); populating a slot is sufficient to make the orchestrator invoke it.
11. **Tests.** One test module per stage 4–7, plus updated integration tests — see [§ 4. Testing Strategy](#4-testing-strategy) in full.
12. **Documentation updates.** Every document in [§ 5. Documentation Updates](#5-documentation-updates), not just the adapter's own design document.
13. **CHANGELOG.** A new `[Unreleased]` entry per completed part — see the Secure Boot Adapter's own four `CHANGELOG.md` entries (models, parser, errors, adapter) for the level of detail expected: what changed, what tests were added, exact test counts, and what remains out of scope for that entry specifically.
14. **IMPLEMENTATION_STATUS update.** Update the adapter's row in [IMPLEMENTATION_STATUS.md § 2. Architecture Components](IMPLEMENTATION_STATUS.md#2-architecture-components) and [§ 4. Platform Adapter Matrix](IMPLEMENTATION_STATUS.md#4-platform-adapter-matrix), and its entry in [§ 5. Host Discovery Status](IMPLEMENTATION_STATUS.md#5-host-discovery-status) if the Composition Root/Host Discovery stages are also complete.
15. **Final quality gates.** `ruff check`, `ruff format --check`, `mypy` (the canonical, package-scoped command — see [IMPLEMENTATION_STATUS.md § 7. Test Status](IMPLEMENTATION_STATUS.md#7-test-status) for exactly what it does and doesn't cover), and `pytest`, all green, before considering any stage complete.

**A note on stages 4–6's ordering:** models, errors, and parser have no dependencies on each other — `parser.py` never imports `errors.py` (a parser raises plain `ValueError`; translating that into a typed exception is the adapter's job, not the parser's), and `errors.py` never imports the domain's own `models.py`. Only the adapter (stage 7) depends on all three. The Secure Boot Adapter was actually built in the order **models → parser → errors → adapter** (see its four separate `CHANGELOG.md` entries), which is the recommended default for consistency across adapters, but a future implementer may build errors before parser without violating any dependency.

## 3. Definition of Done

An adapter is not complete until every item below is true. This extends, and does not replace, [AGENTS.md § Definition of Done](../AGENTS.md#definition-of-done).

- [ ] Design document exists under `docs/`, and its status banner says `Accepted` (not `Proposed`).
- [ ] Any ADR the design required (see [§ 2](#2-adapter-lifecycle) stage 2) is `Accepted`, or the design document explicitly concludes none is required.
- [ ] `models.py`, `errors.py`, `parser.py`, `adapter.py` all exist and match the accepted design.
- [ ] `__init__.py` re-exports every public model, exception, the parser function, and the adapter function.
- [ ] The adapter forces `LANG=C`/`LC_ALL=C` via a full `os.environ` copy (never a two-key `env` dict) before every `CommandRunner.run()` call — see [§ 6. Architecture Rules](#6-architecture-rules).
- [ ] `CommandNotFoundError`/`CommandTimeoutError` propagate unchanged from the adapter (never re-wrapped), matching every adapter built so far.
- [ ] A non-zero exit maps to a recognized-`stderr` "unavailable" subclass or the domain's base error class; a parser `ValueError` is wrapped into the domain's `*ParseError`, chained via `from exc`.
- [ ] Every model, error class, and public function has a docstring naming the design document section it implements.
- [ ] `HostDiscoveryAdapters`/`HostDiscoverySnapshot` slots are wired and concretely typed (if Host Discovery integration is in scope for this change — see [§ 2](#2-adapter-lifecycle) stage 10).
- [ ] `bcs.app.main()`'s composition root binds the new adapter to the shared `CommandRunner`, if Composition Root wiring is in scope.
- [ ] Test modules exist for models, errors, parser, and adapter, at 100% statement and branch coverage on all four files (see [§ 4](#4-testing-strategy)).
- [ ] `cli/tests/fixtures/<domain>/` exists with a populated README (required capture command, layout, naming, and an inventory table) — even if every fixture is still a zero-byte placeholder. No fabricated fixture content, ever.
- [ ] Every document in [§ 5. Documentation Updates](#5-documentation-updates) reflects the change — status banners, package-structure diagrams, dependency diagrams, cross-adapter Open Questions that this adapter resolves.
- [ ] `CHANGELOG.md` has an `[Unreleased]` entry.
- [ ] `IMPLEMENTATION_STATUS.md` reflects the new state.
- [ ] Every internal link and heading anchor touched or added has been verified against the actual target — see [§ 7. Common Mistakes](#7-common-mistakes).
- [ ] `ruff check`, `ruff format --check`, `mypy`, and `pytest` all pass.

## 4. Testing Strategy

Extracted from `test_platform_adapters_{efi,storage,secureboot}_*.py` and each adapter document's own Testing Strategy section (e.g. [SECURE_BOOT_ADAPTER.md § Testing Strategy](SECURE_BOOT_ADAPTER.md#testing-strategy), [STORAGE_ADAPTER.md § Testing Strategy](STORAGE_ADAPTER.md#testing-strategy)).

- **Model tests** — construction, defaults, both alias spellings (`populate_by_name`), every field-level validator, immutability (mutation raises), equality, hashability, and JSON round-tripping (`model_dump(mode="json", by_alias=True)` → `model_validate`). No fixtures, no mocking — direct unit tests. Target: 100% statement and branch coverage on `models.py`.
- **Parser tests** — every recognized line/field pattern individually and combined; permissive handling of unrecognized lines/blank lines/internal whitespace/CRLF endings; the two-tier permissiveness rule in practice (an unrecognized line is silently skipped; a recognized-prefix line with an invalid value is a hard `ValueError` naming the field, the 1-based line number, and the offending line verbatim); a later duplicate overwriting an earlier one, where that's the model's own semantics; empty input still returning a valid, non-error result. Loaded through `fixture_utils.py` against a `tmp_path`-rooted synthetic corpus while the real corpus holds only placeholders (see below) — never an inline string passed straight to the parser, so the loading mechanism itself gets exercised identically to how it will be exercised once real captures exist.
- **Malformed input** is a first-class test category, not an afterthought: truncated/invalid JSON (for JSON-based tools), a missing required field, an unexpected top-level shape. Tested only via the synthetic corpus, never added to the real fixture corpus's inventory — a healthy real system does not produce malformed tool output to capture, so there is nothing legitimate to place there.
- **Adapter tests** — a `FakeCommandRunner` (a small `@dataclass` implementing the `CommandRunner` Protocol, keyed by tool name via `command[0]` when more than one tool is involved, mirroring `test_platform_adapters_storage_adapter.py`'s own shape) stands in for `SubprocessCommandRunner`; never mock it, never touch a real process. Verifies: the exact command and argument list; `check=False`; the locale-forced `env` (`LANG=C`/`LC_ALL=C`, with `PATH` still present — proving the full-`os.environ`-copy requirement, not just the two override keys); the timeout forwarded correctly, including the domain's own default; `CommandNotFoundError`/`CommandTimeoutError` propagating unchanged; each recognized "unavailable" `stderr` pattern mapping to the right subclass; an unrecognized non-zero exit mapping to the base error class; a parser `ValueError` becoming the domain's `*ParseError` with `__cause__`/`from exc` preserved.
- **Protocol conformance** — one direct `isinstance(fake_runner, CommandRunner)` assertion against the `runtime_checkable` Protocol, proving the fake is a legitimate stand-in and not just structurally similar by accident.
- **AST-based import-purity tests** — not a substring search (a module's own docstring may legitimately *discuss* `subprocess`/`CommandRunner` as things it doesn't import). Parse the module's source with `ast.parse`, walk `ast.Import`/`ast.ImportFrom` nodes, and assert the actual import set contains nothing beyond an explicit allow-list. Applied to every parser module and to `HostDiscoveryOrchestrator` itself (see `test_inventory_discovery_orchestrator.py`'s own `test_orchestrator_module_imports_nothing_but_its_own_dependencies`).
- **Coverage expectations** — 100% statement and branch coverage on every file in a completed adapter's package (`models.py`, `errors.py`, `parser.py`, `adapter.py`), verified via the project's own `pytest --cov` configuration. This is not aspirational: every currently-implemented adapter module is already at 100% (see [IMPLEMENTATION_STATUS.md § 7. Test Status](IMPLEMENTATION_STATUS.md#7-test-status)).
- **Integration tests** — two distinct kinds, both established by [`cli/tests/test_host_discovery_pipeline.py`](../cli/tests/test_host_discovery_pipeline.py):
  1. A shared, multi-tool `FakeCommandRunner` binds *every real, currently-implemented* adapter (not lightweight fakes) exactly as `bcs.app.main()` does, and exercises `HostDiscoveryOrchestrator.discover()` end to end — every tool invoked exactly once, `caveats` generated in the exact `"{domain}: {ExceptionType}: {message}"` format, multiple independent failures each isolated correctly.
  2. One test drives the pipeline through the real, unmodified composition root itself: patch only the sanctioned `SubprocessCommandRunner` seam, invoke a real CLI command via `CliRunner`, capture the exact `HostDiscoveryOrchestrator` instance `bcs.app.main()` built, and call `.discover()` on it directly.
  A new adapter should extend `test_host_discovery_pipeline.py`'s fixtures and assertions to include itself, not create a parallel integration test module.
- **Real end-to-end (optional, environment-gated)** — a test that runs the real tool against a real Linux environment, skipped unless the tool is on `PATH` and the platform is Linux, expected to skip in CI. Documented as the plan in every adapter's Testing Strategy section; not actually implemented for any adapter yet — don't claim it exists in a design document without adding it.

## 6. Architecture Rules

Every rule below is already decided — by [ADR-0009](decisions/0009-platform-layer-command-runner.md), [ADR-0010](decisions/0010-efi-adapter-read-only-scope.md), or [ADR-0011](decisions/0011-host-discovery-orchestrator.md) — and restated here only as a checklist, not as a new decision.

- **Adapters execute commands; nothing else does.** `adapter.py` is the *only* module in a domain's package that imports `CommandRunner` or calls `.run()`. `models.py` and `parser.py` must not import `bcs.platform.execution` or `subprocess`, ever — enforced mechanically by an AST-based purity test (see [§ 4](#4-testing-strategy)), not just by convention.
- **Parsers never execute commands.** A parser's signature is `parse_<domain>(text: str) -> <Model>` — never `stdout`, never a `CommandResult`, never anything that presupposes where the text came from. See [EFI_ADAPTER.md § Parser Architecture](EFI_ADAPTER.md#parser-architecture) and every sibling adapter's identical rule.
- **Parsers never infer facts.** A parser reports exactly what its input text says, in a form matching that text one-for-one; it never computes a derived value the source text doesn't state, never guesses at a missing field, never fills in `UNKNOWN`-with-a-best-guess. Where a fact is absent, the model represents that absence (`None`, an `UNKNOWN` enum value) rather than a parser-invented default.
- **Models only represent facts, never policy.** A domain model represents what the firmware/tool/kernel *currently reports* — never what it *should* report, never a comparison against a desired/configured value. That comparison belongs to a consumer (`bcs doctor`), never to the adapter. This is why `SecureBootStatus`, not `SecureBootConfiguration` — see [SECURE_BOOT_ADAPTER.md § Naming Rationale](SECURE_BOOT_ADAPTER.md#naming-rationale) for the full collision-avoidance reasoning.
- **Adapters translate exceptions; parsers raise plain `ValueError`.** A parser's only failure mode is a plain `ValueError` (naming the field, the line number, and the offending text). Only the adapter, which alone knows about `CommandResult`/exit codes/`stderr`, translates that (and a non-zero exit) into the domain's typed `PlatformError` subclasses.
- **The orchestrator isolates `PlatformError`, and only `PlatformError`.** `HostDiscoveryOrchestrator._call_adapter` catches exactly `bcs.platform.errors.PlatformError` (base class or any subclass) into one `caveats` entry; any other exception propagates unmodified and halts the sweep — a genuine bug, never something to swallow. See [HOST_DISCOVERY_ORCHESTRATOR.md § Error Propagation](HOST_DISCOVERY_ORCHESTRATOR.md#error-propagation).
- **The composition root owns dependency injection.** `bcs.app.main()`, and nowhere else, constructs `SubprocessCommandRunner` and `HostDiscoveryAdapters`. No adapter, collector, or command constructs its own `CommandRunner` or reaches for a module-level default/singleton/service locator.
- **The locale policy is forced, always, the same way.** Every adapter builds its child environment as a **full `os.environ.copy()`** with `LANG`/`LC_ALL` overridden to `"C"` — never a two-key dict, since `CommandRunner.run()`'s `env` parameter *replaces* the child's environment entirely rather than merging into it. See [PLATFORM_LAYER.md § Locale Policy](PLATFORM_LAYER.md#locale-policy).
- **Commands are argument lists, never shell strings.** `shell=True` is never passed, anywhere, under any circumstance — see [PLATFORM_LAYER.md § Architectural Rule: Argument Lists Only](PLATFORM_LAYER.md#architectural-rule-argument-lists-only-never-shelltrue).
- **Naming is domain-driven, never tool-driven.** A package is named after the domain concept (`secureboot`, not `mokutil`) so a future backend swap (`efivarfs` reads instead of shelling out to `mokutil`, for instance) touches only `adapter.py` — see [docs/standards/naming-conventions.md § Domain-Driven Naming](standards/naming-conventions.md#domain-driven-naming), first applied in [ADR-0010](decisions/0010-efi-adapter-read-only-scope.md).
- **`bcs.platform` never depends on `bcs.inventory`.** The dependency direction is fixed: `bcs.inventory`/`bcs.inventory.discovery` may depend on `bcs.platform`, never the reverse. Where a Platform Layer domain model shadows an existing `bcs.inventory.models` concept (e.g. `SecureBootState` existing in both places), the Platform Layer's own version is **independently defined**, never imported from `bcs.inventory` — see [SECURE_BOOT_ADAPTER.md § Naming Rationale](SECURE_BOOT_ADAPTER.md#naming-rationale) for why sharing a name deliberately, without an import relationship, is the correct choice here.
- **Adapters are permanently read-only**, unless and until a write-capable variant is proposed as its own, separate design document and (per [§ 2](#2-adapter-lifecycle) stage 2) its own ADR — never a silent extension of an existing read-only adapter. See [EFI_ADAPTER.md § Read-Only Guarantee](EFI_ADAPTER.md#read-only-guarantee).
- **Proportionality is a real constraint, not a formality.** Per [REVIEW.md § 7. A Meta-Concern: Proportionality](../REVIEW.md#7-a-meta-concern-proportionality), a new adapter needs a concrete, cited need (an existing narrow workaround it generalizes, a `SPECIFICATION.md` requirement, a deferred question a sibling document already raised) — not "it would be nice to have" or "the pattern is easy to repeat." Every adapter document's own Purpose section states this motivation explicitly; an adapter design that can't cite one should say so honestly (see [FILESYSTEM_ADAPTER.md § Purpose](FILESYSTEM_ADAPTER.md#purpose)'s own transparent admission that no `SPECIFICATION.md` requirement mandates it) rather than inventing one.

## 7. Common Mistakes

Every item below is a real mistake this project actually made and then fixed — not a hypothetical. Each links to the fix as evidence.

- **Forgetting `__init__.py` exports.** The Secure Boot Adapter's `parser.py` and `adapter.py` both existed for a time before their public functions (`parse_secure_boot_status`, `read_secure_boot_status`) were re-exported from the package's own `__init__.py` — meaning a caller had to reach into a private submodule to use them. Check the package's `__all__` against every public name in `models.py`/`errors.py`/`parser.py`/`adapter.py` explicitly; don't assume a later "wire it in" pass will catch a missing export.
- **Forgetting CHANGELOG entries.** A documentation-only consolidation pass was itself missing a `CHANGELOG.md` entry until a later pass added one retroactively — despite [AGENTS.md § Definition of Done](../AGENTS.md#definition-of-done) requiring one for every `cli/` task. If a change touches `cli/` or `docs/`, it gets an entry in the same change, not "when someone notices."
- **Stale documentation surviving implementation.** Multiple documents (`docs/PLATFORM_LAYER.md`'s own top-level docstring in `cli/src/bcs/platform/__init__.py`, `cli/src/bcs/inventory/discovery/models.py`'s module docstring, `docs/decisions/0008-...`'s and `0009-...`'s and `0010-...`'s own Consequences sections) said "not yet implemented" or "none of this exists" for features that had since shipped — sometimes for two or three completed adapters' worth of time. A design document's own status banner is the easiest place to check first when picking up a task; treat any "not yet implemented" claim as a fact to re-verify against the actual code, not a fact to trust.
- **Broken/stale anchors and cross-references.** `docs/HOST_DISCOVERY_ORCHESTRATOR.md` cross-referenced [ADR-0011](decisions/0011-host-discovery-orchestrator.md) as `Proposed` after it had already been `Accepted`; [`docs/SECURE_BOOT_ADAPTER.md`](SECURE_BOOT_ADAPTER.md) and [`docs/FILESYSTEM_ADAPTER.md`](FILESYSTEM_ADAPTER.md) each independently reserved the hypothetical ADR number **"ADR-0012"** for two different, unrelated future decisions — a collision that would only have surfaced when someone actually tried to claim the number. Verify every link and anchor you touch or add against the real target; don't assume a cross-reference written months ago is still accurate, and don't guess at an ADR number a sibling document might already be reserving.
- **Locale handling done as a two-key dict instead of a full copy.** `CommandRunner.run()`'s `env` parameter *replaces* the child's environment; passing only `{"LANG": "C", "LC_ALL": "C"}` drops `PATH` and breaks execution entirely, not just locale. Every adapter's own test suite asserts `"PATH" in call["env"]` specifically to catch this.
- **A parser performing interpretation instead of structural parsing.** A parser recognizing `SecureBoot enabled`/`SecureBoot disabled` is structural parsing; a parser deciding *whether that's good* is interpretation, and belongs nowhere in this codebase's adapters — see [§ 6. Architecture Rules](#6-architecture-rules)'s "models only represent facts" rule. Watch for this specifically at review time: a percentage-used field, a `"healthy"`/`"full"` classification, or a "primary disk" inference are all instances of the same mistake, and [FILESYSTEM_ADAPTER.md § Scope Guarantee](FILESYSTEM_ADAPTER.md#scope-guarantee) names several of them explicitly as things that were considered and rejected.
- **Adding heuristics where an explicit "don't know" is correct.** The Storage Adapter never infers a primary/system/boot disk from its device tree — a real, previously-considered candidate simplification, rejected specifically because it would require guessing at a fact the tools don't state. Where a fact isn't directly observable, the correct model value is `None`/`UNKNOWN`, never a best-effort guess presented as if it were observed.
- **Introducing an upward dependency from `bcs.platform` into `bcs.inventory`.** It would have been the more obvious move to import `bcs.inventory.models.SecureBootState` directly into `bcs.platform.adapters.secureboot.models` rather than redefining an identical enum — but that creates exactly the layering violation [§ 6. Architecture Rules](#6-architecture-rules) forbids. When a Platform Layer domain model appears to duplicate an existing Host Inventory concept, that's very likely intentional, not an oversight to "clean up" by importing across the boundary.
- **A test-count claim in `CHANGELOG.md` that doesn't match reality.** One historical entry claimed "32 new tests" for a module that, both by direct count and by `pytest --collect-only`, actually added 30. Caught by audit, not fixed retroactively (`CHANGELOG.md` entries are a historical record, not a living document — see [AGENTS.md](../AGENTS.md)'s general git-history-is-authoritative stance). Get the count right the first time by running the actual collection command before writing the entry, not by estimating.
- **Treating a fixture corpus category's own README as someone else's job.** More than once, a fixture category README (`cli/tests/fixtures/<domain>/README.md`) still said "no adapter designed yet" well after that adapter's design was accepted, because populating that README felt like a separate, later task. It is not — see [§ 5. Documentation Updates](#5-documentation-updates); the fixtures README is part of the same change as the design document reaching `Accepted`.

## 8. New Adapter Checklist

A concise, mechanical sequence for implementing a new Platform Layer adapter. Follow in order; do not skip a step because a later one looks more urgent.

1. Read [AGENTS.md](../AGENTS.md), [PLATFORM_LAYER.md](PLATFORM_LAYER.md), and the three existing adapter documents ([EFI_ADAPTER.md](EFI_ADAPTER.md), [STORAGE_ADAPTER.md](STORAGE_ADAPTER.md), [SECURE_BOOT_ADAPTER.md](SECURE_BOOT_ADAPTER.md)) before writing anything.
2. Confirm a concrete, citable motivation exists (an existing narrow workaround, a `SPECIFICATION.md` requirement, or a deferred question a sibling document already raised) — per [§ 6. Architecture Rules](#6-architecture-rules)'s proportionality rule. If none exists, say so and stop; do not invent one.
3. Write `docs/<DOMAIN>_ADAPTER.md` using the section order in [§ 5. Documentation Updates](#5-documentation-updates). Status: `Proposed, pending approval`. No code yet.
4. Decide whether a new ADR is required (almost certainly not — see [§ 2](#2-adapter-lifecycle) stage 2) and record that conclusion in the design document's own ADR Recommendation section.
5. Get the design reviewed and accepted; update the status banner to `Accepted` only after that happens, and only with explicit user/maintainer approval to begin implementation — design acceptance and implementation approval are two separate approvals, per [AGENTS.md § Workflow](../AGENTS.md#workflow).
6. Implement `models.py`, then `parser.py`, then `errors.py`, then `adapter.py` (see [§ 2](#2-adapter-lifecycle)'s note on this ordering), each with its own test module reaching 100% statement and branch coverage before moving to the next file.
7. Update `__init__.py` to re-export every public name from all four files.
8. Create `cli/tests/fixtures/<domain>/README.md` with the required capture command, layout, naming convention, and an inventory table of zero-byte placeholder scenarios — never fabricated content.
9. Wire the new `read_*` function into `HostDiscoveryAdapters` at `bcs.app.main()`'s composition root, sharing the existing `SubprocessCommandRunner` instance.
10. Narrow the corresponding `HostDiscoveryAdapters`/`HostDiscoverySnapshot` slot from `object`-typed to the adapter's own concrete model, in `cli/src/bcs/inventory/discovery/models.py`.
11. Extend `cli/tests/test_host_discovery_pipeline.py` to include the new adapter in its shared multi-tool `FakeCommandRunner` and its assertions.
12. Update every document in [§ 5. Documentation Updates](#5-documentation-updates) — not just the new adapter's own design document.
13. Add a `CHANGELOG.md` entry (or one per completed part, matching [§ 2](#2-adapter-lifecycle) stage 13's level of detail).
14. Update `docs/IMPLEMENTATION_STATUS.md`.
15. Verify every internal link and heading anchor touched or added.
16. Run `ruff check`, `ruff format --check`, `mypy`, and `pytest`; confirm all four are green before calling any part of this done.
17. Stop. Do not begin wiring the adapter into `bcs doctor`/`bcs inventory` or folding its output into `HostInventory`'s own schema unless that was explicitly part of the requested scope — both are established, separate, later steps for every adapter implemented so far (see [IMPLEMENTATION_STATUS.md § 5. Host Discovery Status](IMPLEMENTATION_STATUS.md#5-host-discovery-status)).
