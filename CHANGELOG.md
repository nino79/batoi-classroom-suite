# Changelog

All notable changes to Batoi Classroom Suite (BCS) will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html). BCS is still overwhelmingly documentation-first — Boot Manager, Builder, and Deploy remain unimplemented — so version numbers largely track the maturity of the *specification*; the `bcs` CLI framework is the first exception with real, tested code.

## [Unreleased]

### Added

- Full documentation set: `ARCHITECTURE.md`, `SPECIFICATION.md`, `ROADMAP.md`, `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`, `AGENTS.md`, `CLAUDE.md`.
- `docs/` tree: architecture deep-dives, component specifications, Architecture Decision Records, coding/Bash/Markdown/naming standards, development and release process docs, contributor guides, and glossary.
- `.github/` issue templates, pull request template, label taxonomy, and Discussions category documentation.
- Placeholder READMEs describing the intended purpose of `boot-manager/`, `builder/`, `deploy/`, `scripts/`, `tools/`, `tests/`, and `assets/`.
- `REVIEW.md`: an independent, critical architecture review of the documentation set, with a prioritized punch list.
- Unified BCS configuration system: `config/schema.yaml` (normative JSON Schema, Kubernetes-style `apiVersion`/`kind`/`metadata`/`spec` envelope), `config/examples/default.yaml` (reference instance), and `docs/CONFIGURATION.md` (full field reference). Resolves the "recipe/manifest" naming ambiguity and the previously-undefined Builder recipe format (`BLD-001`) — see [ADR-0005](docs/decisions/0005-yaml-as-unified-configuration-format.md).

- `docs/CLI.md`: complete design of `bcs`, the unified command-line interface — command tree, global options, exit codes, logging/verbosity, color output, progress reporting, configuration loading, validation flow, and the git-style plugin system. New `CLI-001`–`CLI-014` requirements in `SPECIFICATION.md §2.4`. See [ADR-0006](docs/decisions/0006-bcs-unified-cli-architecture.md).
- `cli/`: the `bcs` CLI framework, implemented in Python (Typer, Rich, Pydantic, PyYAML) per [ADR-0007](docs/decisions/0007-python-for-the-bcs-cli.md) — global options, structured logging, `NO_COLOR`-aware output, the ClassroomConfig loader/override/validation pipeline (Pydantic models mirroring `config/schema.yaml`), git-style plugin dispatch, and the `version`/`doctor`/`validate` commands with placeholder check logic. `build`/`install`/`deploy`/`backup`/`restore`/`update`/`config` are registered as stubs.
- `.pre-commit-config.yaml` and `.github/workflows/ci.yml`: lint (Ruff), type-check (mypy), and test (pytest, Python 3.12/3.13) gates for `cli/`, plus a `bcs` smoke-test job.
- **Host Inventory subsystem** (`bcs.inventory`): immutable, extensible Pydantic models (`HostInventory` and its sections — firmware/Secure Boot, storage, network, identity, OS, CPU, memory, tooling), stdlib-only host collectors, and the `bcs inventory` command. This is BCS's single source of truth describing the current machine, consumed identically by `bcs doctor` (refactored to evaluate pass/fail against these same facts instead of probing the host a second time), and — once implemented — by Boot Manager, Builder, and Deploy, plus a future REST API/Web UI. Carries its own `bcs-inventory/v1alpha1` schema version, independent of `bcs-cli/v1alpha1` and `bcs/v1alpha1`. New `CLI-015` requirement in `SPECIFICATION.md §2.4`. `HostIdentity` narrows (does not resolve) the open `deploy.maintenanceRequests.machineIdentity` question. 209 pytest cases; Ruff, mypy (strict), pre-commit, and GitHub Actions CI all pass.
- `bcs.model_utils`: the `x-`-prefixed-extra-key validator shared by `bcs.config.models` and `bcs.inventory.models`, extracted so both domains can't drift on what counts as a valid extension key.
- `docs/HOST_INVENTORY.md`: a complete design proposal for the Host Inventory subsystem — package structure, Pydantic model diagram, per-class/module responsibilities, dependency graph, sequence diagrams, interaction with the CLI/future REST API/future Web UI, serialization strategy, testing strategy, a current-implementation-status-vs-proposal reconciliation table, and a numbered list of proposed changes requiring approval (a `caveats` field, a checked-in JSON Schema artifact, a golden-file schema-regression test). Supported by [ADR-0008](docs/decisions/0008-host-inventory-ports-and-adapters.md), formalizing the subsystem's ports-and-adapters split, immutability, and JSON-as-canonical-format decisions. **This is documentation only — no `.py`, `pyproject.toml`, or test files were changed**.
- [ADR-0008](docs/decisions/0008-host-inventory-ports-and-adapters.md) accepted: the ports-and-adapters split, immutability, and JSON-as-canonical-format decisions for the Host Inventory subsystem are now in effect. Approving the architecture does not, by itself, approve the individual items in `docs/HOST_INVENTORY.md`'s Proposed Changes Requiring Approval list (the `caveats` field, the checked-in JSON Schema artifact, the golden-file regression test) — those remain open. **Documentation only — no code changed.**
- `EfiSystemPartition` and `UsbStorageDevice` added to the Host Inventory schema, per an amendment to [ADR-0008](docs/decisions/0008-host-inventory-ports-and-adapters.md#amendment-efi-system-partition-and-usb-storage) and new requirement `CLI-016` in `SPECIFICATION.md §2.4`. **Documentation only — no code changed** (`docs/HOST_INVENTORY.md`, `docs/decisions/0008-...`, `SPECIFICATION.md`, `docs/CLI.md`, `ARCHITECTURE.md`).
- Implemented the two new Host Inventory facts: `collect_efi_system_partition()` (probes `/boot/efi` via `/proc/mounts`, resolves the parent disk and filesystem UUID, reports mount/size/free-space state) and `collect_usb_storage()` (enumerates removable, USB-attached block devices via `/sys/block` + sysfs `removable`/device-path heuristics, excluding internal NVMe and non-storage USB peripherals). Both are wired into `collect_host_inventory()`, `bcs inventory`'s text/JSON/YAML rendering, and two new `bcs doctor` checks (`esp`, `usb-storage`). 25 new/updated tests (234 total); Ruff, mypy (strict), and pytest all pass. No Boot Manager, Builder, or Deploy logic was added.
- `docs/PLATFORM_LAYER.md`: a design proposal for the Platform Layer — the `bcs.platform` package that will centralize every OS process execution in `cli/` behind a `CommandRunner` interface, so business/command code never calls `subprocess` directly. Covers package structure, the `CommandRunner` API, the `CommandResult` model, the `PlatformError` exception hierarchy, per-call vs. whole-invocation timeout handling, logging strategy, dependency injection (a proposed `RuntimeContext.command_runner` field), a three-layer testing strategy, and how future adapters (`efibootmgr`, `lsblk`, `blkid`, `mount`, `rsync`) will depend on it. Supported by [ADR-0009](docs/decisions/0009-platform-layer-command-runner.md) (status: `Proposed`) and new requirement `NFR-008` in `SPECIFICATION.md §3` (reusing the existing `NFR-` prefix rather than introducing a new one). **This is documentation only — no `.py`, `pyproject.toml`, or test files were changed**; it is pending explicit approval before any implementation.
- [ADR-0009](docs/decisions/0009-platform-layer-command-runner.md) accepted, with five amendments incorporated directly into the design: the implementation module renamed `runner.py` → `execution.py`; `CommandResult` extended with `command`, `exitCode`, `duration`, `startedAt`, `finishedAt`, `workingDirectory`, and a computed `success` property; "commands are argument lists, `shell=True` is never used" promoted to its own named architectural rule; `docs/standards/coding-standards.md` updated with an explicit, tool-named prohibition (`subprocess.run()`/`subprocess.Popen()`/equivalent outside `bcs.platform`); and the `cli/pyproject.toml` Ruff-scoping plan (`S603`/`S607` narrowed to `bcs.plugins` and `bcs.platform.execution` only) confirmed as approved design. **Documentation only — no code changed** (`docs/PLATFORM_LAYER.md`, `docs/decisions/0009-...`, `docs/standards/coding-standards.md`, `ARCHITECTURE.md`, `docs/CLI.md`); implementation remains gated on a further explicit go-ahead.
- **Platform Layer implemented (Platform-001, Parts 1–3)**: `bcs.platform.models.CommandResult` (frozen, JSON-serializable, computed `success` property, cross-field validators for the `exit_code`/`timed_out` and `started_at`/`finished_at` invariants); `bcs.platform.errors` (`PlatformError`, `CommandNotFoundError`, `CommandTimeoutError`, `CommandExecutionError` — deliberately independent of `bcs.errors.BcsError`); `bcs.platform.execution` (`CommandRunner`, a `runtime_checkable` `Protocol`, and `SubprocessCommandRunner`, wrapping `subprocess.run` with `shell=True` never used, argument lists only, captured stdout/stderr, per-call timeout enforcement). No adapters, no `RuntimeContext` integration, no business logic changed. Ruff, mypy (strict), and pytest all pass with 100% coverage on all three new modules.
- **`CommandRunner` wired into `RuntimeContext` (Platform-001, Part 4)**: `bcs.app.main()` now constructs exactly one `SubprocessCommandRunner` per invocation and injects it as `RuntimeContext.command_runner` — dependency injection only, no existing command's behavior changed, no adapters, no service locator or module-level singleton. `docs/PLATFORM_LAYER.md`'s Dependency Injection section and dependency-graph diagram updated with the ownership/lifecycle detail. 299 pytest cases across `cli/`; Ruff, mypy (strict), and pytest all pass.
- `docs/EFI_ADAPTER.md`: a design proposal for the EFI Adapter — the first **Host Discovery** adapter, a read-only wrapper around `efibootmgr` built on the Platform Layer. Covers package structure (an internal `bcs.platform.adapters.efibootmgr` subpackage: `models.py`/`parser.py`/`adapter.py`/`errors.py`), the `EfiBootConfiguration`/`BootEntry` Pydantic models, a pure-function parser architecture, adapter responsibilities (locale-forced `LC_ALL=C`/`LANG=C` invocation, `check=False`, explicit timeout), error mapping (`EfibootmgrError`/`EfibootmgrUnavailableError`/`EfibootmgrParseError`, extending `PlatformError`), supported `efibootmgr` versions (Ubuntu 24.04 LTS as the pinned reference, best-effort elsewhere), a testing strategy using real captured `efibootmgr -v` output as fixtures, and future extensibility. States a hard, permanent read-only guarantee: never modifies EFI variables, never changes boot order, never installs rEFInd. Supported by [ADR-0010](docs/decisions/0010-efi-adapter-read-only-scope.md) (status: `Proposed`). **This is documentation only — no `.py` files were changed**; it is pending explicit approval before any implementation.
- [ADR-0010](docs/decisions/0010-efi-adapter-read-only-scope.md) accepted, with six amendments incorporated directly into the design: the adapter package renamed `bcs.platform.adapters.efibootmgr` → `.efi` (domain-named, not tied to the current backend tool); models renamed `EfiBootConfiguration` → `FirmwareBootConfiguration` (avoiding a real naming collision with Boot Manager's own `spec.bootManager.menu` "boot configuration") and exceptions `Efibootmgr*Error` → `FirmwareBoot*Error`; the pure parser's independence from the execution layer made an explicit, load-bearing property (`text: str` in, never imports `CommandRunner`/`subprocess`, never knows its input's provenance); a new **project-wide architectural rule** — "packages, adapters, and Pydantic models are named after the business/domain concept, never the implementation technology" — added to `docs/standards/naming-conventions.md` and `docs/standards/coding-standards.md`, and cross-referenced from `ARCHITECTURE.md §5`; the `LANG=C`/`LC_ALL=C` locale requirement promoted from an EFI-specific detail to a Platform Layer-wide rule in `docs/PLATFORM_LAYER.md`; and command-execution metadata evaluated and concluded *not* to belong on `FirmwareBootConfiguration` (with a reusable `CommandMetadata`/envelope concept documented as a deferred, not-yet-necessary idea for future adapters). Explicitly confirmed not to affect any already-implemented Platform-001 public name (`CommandResult`, `PlatformError`, `CommandRunner`/`SubprocessCommandRunner`, `RuntimeContext.command_runner`). **Documentation only — no code changed** (`docs/EFI_ADAPTER.md`, `docs/decisions/0010-...`, `docs/standards/naming-conventions.md`, `docs/standards/coding-standards.md`, `docs/PLATFORM_LAYER.md`, `ARCHITECTURE.md`, `docs/glossary.md`, `docs/architecture/boot-manager.md`); implementation remains gated on a further explicit go-ahead.
- **EFI Adapter domain models implemented (Host Discovery, models only)**: `bcs.platform.adapters.efi.models.BootEntry` and `.FirmwareBootConfiguration` — frozen, JSON-serializable Pydantic models with full type annotations and docstrings, per `docs/EFI_ADAPTER.md#pydantic-models`. Validates the four-hex-digit boot-number format (`boot_number`, `current_boot_number`, `boot_next`, every `boot_order` entry) and rejects `entries` with duplicate `boot_number` values. No parsing logic, no `subprocess`, no `CommandRunner`, no adapter, no CLI integration — `parser.py`/`adapter.py`/`errors.py` remain unimplemented. 38 new tests (`cli/tests/test_platform_adapters_efi_models.py`) covering construction, validation, immutability, equality, hashability, and JSON serialization/deserialization round-tripping; Ruff, mypy (strict), and pytest all pass with 100% coverage on the new module.

### Changed

- `docs/standards/naming-conventions.md`: configuration keys are `camelCase` (Kubernetes/Compose convention), superseding earlier `snake_case` guidance that predated the configuration format being designed; registered the new `CLI-NNN` requirement ID prefix.
- `docs/glossary.md`: "Recipe" now points to the concrete `ClassroomConfig` format instead of describing an undefined artifact; "manifest" is retired as a synonym.
- `ARCHITECTURE.md`: added `§8 Operator Interface`, describing `bcs` as the single entry point into all three components.
- `docs/CLI.md`: `Implementation Notes` now describes the actual Python layout under `cli/`, superseding the originally-planned Bash layout.
- `AGENTS.md`, `docs/repository-organization.md`: updated to reflect `cli/` as an implemented, tested exception to the documentation-only phase.
