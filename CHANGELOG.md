# Changelog

All notable changes to Batoi Classroom Suite (BCS) will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html). BCS is still overwhelmingly documentation-first — Boot Manager, Builder, and Deploy remain unimplemented — so version numbers largely track the maturity of the *specification*; the `bcs` CLI framework is the first exception with real, tested code.

This is BCS's single source of truth describing the current machine, consumed identically by `bcs doctor` (refactored to evaluate pass/fail against these same facts instead of probing the host a second time), and — once implemented — by Boot Manager, Builder, and Deploy, plus a future REST API/Web UI. Carries its own `bcs-inventory/v1alpha1` schema version, independent of `bcs-cli/v1alpha1` and `bcs/v1alpha1`. New `CLI-015` requirement in `SPECIFICATION.md §2.4`. `HostIdentity` narrows (does not resolve) the open `deploy.maintenanceRequests.machineIdentity` question. 209 pytest cases; Ruff, mypy (strict), pre-commit, and GitHub Actions CI all pass.

Covers package structure, the `CommandRunner` API, the `CommandResult` model, the `PlatformError` exception hierarchy, per-call vs. whole-invocation timeout handling, logging strategy, dependency injection (a proposed `RuntimeContext.command_runner` field), a three-layer testing strategy, and how future adapters (`efibootmgr`, `lsblk`, `blkid`, `mount`, `rsync`) will depend on it. - `docs/EFI_ADAPTER.md`: a design proposal for the EFI Adapter — the first **Host Discovery** adapter, a read-only wrapper around `efibootmgr` built on the Platform Layer. Covers package structure (an internal `bcs.platform.adapters.efibootmgr` subpackage: `models.py`/`parser.py`/`adapter.py`/`errors.py`), the `EfiBootConfiguration`/`BootEntry` Pydantic models, a pure-function parser architecture, adapter responsibilities (locale-forced `LC_ALL=C`/`LANG=C` invocation, `check=False`, explicit timeout), error mapping (`EfibootmgrError`/`EfibootmgrUnavailableError`/`EfibootmgrParseError`, extending `PlatformError`), supported `efibootmgr` versions (Ubuntu 24.04 LTS as the pinned reference, best-effort elsewhere), a testing strategy using real captured `efibootmgr -v` output as fixtures, and future extensibility. Supported by [ADR-0010](docs/decisions/0010-efi-adapter-read-only-scope.md) (status: `Proposed`). **This is documentation only — no `.py` files were changed**; it is pending explicit approval before any implementation.
- [ADR-0010](docs/decisions/0010-efi-adapter-read-only-scope.md) accepted, with six amendments incorporated directly into the design: the adapter package renamed `bcs.platform.adapters.efibootmgr` → `.efi` (domain-named, not tied to the current backend tool); models renamed `EfiBootConfiguration` → `FirmwareBootConfiguration` (avoiding a real naming collision with Boot Manager's own `spec.bootManager.menu` "boot configuration") and exceptions `Efibootmgr*Error` → `FirmwareBoot*Error`; the pure parser's independence from the execution layer made an explicit, load-bearing property (`text: str` in, never imports `CommandRunner`/`subprocess`, never knows its input's provenance); a new **project-wide architectural rule** — "packages, adapters, and Pydantic models are named after the business/domain concept, never the implementation technology" — added to `docs/standards/naming-conventions.md` and `docs/standards/coding-standards.md`, and cross-referenced from `ARCHITECTURE.md §5`; the `LANG=C`/`LC_ALL=C` locale requirement promoted from an EFI-specific detail to a Platform Layer-wide rule in `docs/PLATFORM_LAYER.md`; and command-execution metadata evaluated and concluded *not* to belong on `FirmwareBootConfiguration` (with a reusable `CommandMetadata`/envelope concept documented as a deferred, not-yet-necessary idea for future adapters). Explicitly confirmed not to affect any already-implemented Platform-001 public name (`CommandResult`, `PlatformError`, `CommandRunner`/`SubprocessCommandRunner`, `RuntimeContext.command_runner`). **Documentation only — no code changed** (`docs/EFI_ADAPTER.md`, `docs/decisions/0010-...`, `docs/standards/naming-conventions.md`, `docs/standards/coding-standards.md`, `docs/PLATFORM_LAYER.md`, `ARCHITECTURE.md`, `docs/glossary.md`, `docs/architecture/boot-manager.md`); implementation remains gated on a further explicit go-ahead.
- **EFI Adapter fully implemented (Host Discovery)**: `bcs.platform.adapters.efi` — `models.py` (`BootEntry`, `FirmwareBootConfiguration`), `parser.py` (`parse_efibootmgr_output`), `errors.py` (`FirmwareBootError`, `FirmwareBootUnavailableError`, `FirmwareBootParseError`), and `adapter.py` (`read_firmware_boot_configuration`). Frozen, JSON-serializable Pydantic models with full type annotations and docstrings, per `docs/EFI_ADAPTER.md#pydantic-models`. Validates the four-hex-digit boot-number format (`boot_number`, `current_boot_number`, `boot_next`, every `boot_order` entry) and rejects `entries` with duplicate `boot_number` values. The pure-function parser is independent of the execution layer (`text: str` in, never imports `CommandRunner`/`subprocess`). - **Storage Adapter parser implemented (Host Discovery, parser only)**: `bcs.platform.adapters.storage.parser.parse_storage_topology(lsblk_output, blkid_output, findmnt_output)` — a pure function composing `lsblk -J -b`, `blkid -p -o json`, and `findmnt -J` JSON text into an immutable `StorageConfiguration`. Builds the device tree from `lsblk`, annotates partitions with `blkid`'s filesystem/GPT facts, and cross-references `findmnt`'s mount table onto the tree and onto each partition's nested `FilesystemInfo`; never infers a primary/system/boot disk or an installation target. Raises plain `ValueError` for malformed structure (not `StorageParseError`) and lets `pydantic.ValidationError` propagate for model invariants, mirroring `bcs.platform.adapters.efi.parser` exactly — translating to `StorageParseError` remains `adapter.py`'s job, still unimplemented. 30 new tests (`cli/tests/test_platform_adapters_storage_parser.py`) against a synthetic, `tmp_path`-rooted corpus (the real `cli/tests/fixtures/storage/` corpus gained nine zero-byte placeholders — three scenarios × three tools — but no real captures yet) plus an AST-based import-purity check. No `adapter.py`, no `RuntimeContext`/`bcs.inventory` integration, no Host Discovery Orchestrator wiring. Ruff, mypy (strict), and pytest all pass.
- **Storage Adapter fully implemented (Host Discovery, adapter — final part)**: `bcs.platform.adapters.storage.adapter.read_storage_topology(runner, *, timeout_seconds=10.0)` — the orchestration layer, the only module in the package that calls `CommandRunner.run()`. Executes `lsblk -J -b`, `blkid -p -o json`, and `findmnt -J` in sequence (stopping at the first failure), with `LANG=C`/`LC_ALL=C` forced and `check=False` on every call, then hands their stdout to `parse_storage_topology`. Contains no parsing logic. `CommandNotFoundError`/`CommandTimeoutError` propagate unchanged (matching the EFI adapter's own precedent exactly, and consistent with `StorageUnavailableError`'s own "tool present but environment unavailable" docstring); a non-zero exit maps to `StorageUnavailableError` (recognised "unavailable" `stderr`) or `StorageError` (anything else); a parser `ValueError` is wrapped as `StorageParseError`, chained via `from exc`. `docs/STORAGE_ADAPTER.md`'s Error Mapping section is amended accordingly, documenting this as a deliberate deviation from the design doc's earlier, pre-EFI-precedent "tool not found → `StorageUnavailableError`" wording. 27 new tests (`cli/tests/test_platform_adapters_storage_adapter.py`), a `FakeCommandRunner` stand-in keyed by tool name, zero real subprocess execution; `adapter.py` at 100% coverage. This completes the Storage Adapter as designed — `models.py`, `errors.py`, `parser.py`, and `adapter.py` are all implemented. No `RuntimeContext`/`bcs.inventory` integration, no Host Discovery Orchestrator wiring, no Secure Boot/Filesystem adapters. Ruff, mypy (strict), and pytest all pass.
- **Host Discovery Orchestrator — data-holding types implemented (Part 1)**: `bcs.inventory.discovery.models.HostDiscoveryAdapters` (a frozen `dataclass`, one optional `Callable` slot per Discovery domain — `efi`, `storage`, `secure_boot`, `filesystem`, `network`, `cpu`, `memory`, `tpm` — explicit named slots, no dynamic registry, no plugin mechanism, no string-keyed dict) and `bcs.inventory.discovery.models.HostDiscoverySnapshot` (a frozen, `camelCase`-aliased Pydantic model mirroring those same eight domains plus `caveats`). Two corrections made during implementation, both documented in `docs/HOST_DISCOVERY_ORCHESTRATOR.md`: `HostDiscoveryAdapters.network` is typed `Callable[[], list[NetworkInterface]]`, not `tuple[...]`, matching `collectors.collect_network`'s real return type (this project's own architecture review of the design, finding 6); and a `HostDiscoverySnapshot` is hashable only when `network` is empty — any populated `NetworkInterface` breaks it via its own `ip_addresses: list[str]` field, regardless of whether that list is empty, the same limitation `HostInventory` itself already has. [ADR-0011](docs/decisions/0011-host-discovery-orchestrator.md) and `docs/HOST_DISCOVERY_ORCHESTRATOR.md` accepted as part of this change. 18 new tests (`cli/tests/test_inventory_discovery_models.py`); `discovery/models.py` at 100% coverage. No `orchestrator.py`, no composition-root wiring, no `RuntimeContext`/`bcs.inventory` integration, no Secure Boot/Filesystem business logic. Ruff, mypy (strict), and pytest all pass.
- **Host Discovery Orchestrator — coordination logic implemented (Part 2)**: `bcs.inventory.discovery.orchestrator.HostDiscoveryOrchestrator`, constructed with a `HostDiscoveryAdapters` bundle, with one public method, `discover() -> HostDiscoverySnapshot`. Calls every configured (non-`None`) adapter slot exactly once, in the fixed order `efi`/`storage`/`secure_boot`/`filesystem`/`network`/`cpu`/`memory`/`tpm`, via a small `[T]`-generic private helper (`_call_adapter`) rather than a loop over a dict of field names. A `PlatformError` (base class or any subclass) raised by one adapter leaves that domain's field `None`/empty and appends one `"{domain}: {ExceptionType}: {message}"` entry to `caveats`; it never stops the remaining domains. Any other exception propagates unmodified — same instance, no wrapping — and halts the sweep immediately, so domains after the failing one are never called. The `network` slot's `list[NetworkInterface]` result is converted to a `tuple` on the snapshot, and stays `()` (never `None`) whether the slot is unset or fails. Never imports `subprocess` or `bcs.platform.execution.CommandRunner` — verified by an AST-based purity test, not just by reading the code. 17 new tests (`cli/tests/test_inventory_discovery_orchestrator.py`) using lightweight fake adapter callables (a small `_CountingAdapter[T]` plus plain functions for the execution-order test) — no `FakeCommandRunner`, no subprocess mocking of any kind; `discovery/orchestrator.py` at 100% coverage. This completes the orchestrator's own coordination logic as designed. No composition-root wiring, no `RuntimeContext`/`bcs.inventory` integration, no adapter implementation changes. Ruff, mypy (strict), and pytest all pass.
- **Host Discovery Orchestrator — Host Inventory integration implemented (Part 3)**: `bcs.inventory.service.collect_host_inventory()` gains an optional `orchestrator: HostDiscoveryOrchestrator | None = None` parameter. Omitted (the default), behaviour is byte-for-byte identical to before this parameter existed. Given, `orchestrator.discover()` is called exactly once; its `HostDiscoverySnapshot` supplies `network` unconditionally (an empty list is already a valid `HostInventory.network` value) and `cpu`/`memory` whenever the snapshot's own `cpu`/`memory` fields are not `None`, falling back to the same `collectors.collect_cpu()`/`collect_memory()` call otherwise — a refinement beyond `docs/HOST_DISCOVERY_ORCHESTRATOR.md`'s original "satisfied from the snapshot instead" wording, needed because `HostInventory.cpu`/`memory` are *required* fields while the snapshot's are `Optional`, so an unconfigured or `PlatformError`-isolated adapter slot must not be passed through as `None`. Every other `HostInventory` section (`identity`, `firmware`, `operating_system`, `efi_system_partition`, `storage`, `usb_storage`, `tooling`) is unaffected either way. No `try`/`except` wraps `orchestrator.discover()` — any non-`PlatformError` exception (the only kind that can reach here, since `PlatformError` is already isolated inside the orchestrator) propagates unmodified, matching how no existing collector call is wrapped. Per [ADR-0011](docs/decisions/0011-host-discovery-orchestrator.md) Decision point 6, this does **not** add `firmwareBootConfiguration`/`storageTopology`/`secureBoot`/`filesystem`/`tpm`/`caveats` as new `HostInventory` fields — that remains a separate, not-yet-accepted follow-up. 11 tests in `cli/tests/test_inventory_service.py` (2 pre-existing, unchanged, plus 9 new) covering the no-orchestrator/omitted-orchestrator/explicit-`None` cases, snapshot-sourced values overriding collectors, the other seven sections' independence, exactly-once invocation, the `cpu`/`memory` fallback (both the unset-slot and the isolated-`PlatformError` cases), `network`'s no-fallback behaviour, and unexpected-exception propagation; `service.py` at 100% statement and branch coverage. No EFI/Storage adapter, `RuntimeContext`, CLI command, Boot Manager, Builder, or Deploy changes. Ruff, mypy (strict), and pytest all pass; full suite (654 tests) green.
- [docs/SECURE_BOOT_ADAPTER.md](docs/SECURE_BOOT_ADAPTER.md) accepted: the design for the Secure Boot Adapter, the third Host Discovery adapter, wrapping `mokutil --sb-state` — no new ADR required (`docs/SECURE_BOOT_ADAPTER.md#adr-recommendation` concludes every architectural mechanism it uses was already decided by ADR-0008/0009/0010). **Documentation only — no code changed.**
- **Secure Boot Adapter domain models implemented (Host Discovery, models only — Part 1)**: `bcs.platform.adapters.secureboot.models.SecureBootState` (an independently defined `StrEnum` — `ENABLED`/`DISABLED`/`UNSUPPORTED`/`UNKNOWN` — deliberately not imported from `bcs.inventory.models.SecureBootState`, to keep `bcs.platform` from depending upward into `bcs.inventory`) and `.SecureBootStatus` (frozen, JSON-serializable Pydantic model: `state`, `setup_mode: bool | None`, `raw_text`). Named `SecureBootStatus`, not `SecureBootConfiguration`, to avoid colliding with the existing, policy-flavored `spec.security.secureBoot.mode`. No cross-field validation — the model has no collection field for a uniqueness constraint to apply to; validation is limited to intrinsic Pydantic field typing (`state` must be a recognized enum value). Directory named `secureboot` (one word), matching the domain category already reserved in the fixture corpus (`cli/tests/fixtures/secureboot/`) and the sibling `firmware`/`storage`/`filesystem` names. No `parser.py`, `adapter.py`, `errors.py`, `RuntimeContext` integration, or Host Discovery Orchestrator wiring. 18 new tests (`cli/tests/test_platform_adapters_secureboot_models.py`) covering construction, defaults, every enum value, `populate_by_name` aliases, validation (invalid state, extra fields, missing required fields), immutability, equality, hashability, JSON round-tripping, and independence from `bcs.inventory.models.SecureBootState`; `secureboot/models.py` and `secureboot/__init__.py` at 100% coverage. No EFI/Storage adapter, `RuntimeContext`, Host Discovery integration, CLI command, Boot Manager, Builder, or Deploy changes. Ruff, mypy (strict), and pytest all pass; full suite (672 tests) green.
- **Secure Boot Adapter parser implemented (Host Discovery, parser only — Part 2)**: `bcs.platform.adapters.secureboot.parser.parse_secure_boot_status(text: str) -> SecureBootStatus` — a pure function parsing `mokutil --sb-state` output line by line. Recognizes the literal `SecureBoot enabled`/`SecureBoot disabled` and `SetupMode enabled`/`SetupMode disabled` lines (tolerating extra internal whitespace); any other line is silently ignored, and text with no recognized line at all still returns a valid `SecureBootStatus` (`state=UNKNOWN`, `setup_mode=None`) rather than an error. A line starting with the literal `SecureBoot `/`SetupMode ` prefix whose value is neither `enabled` nor `disabled` raises a plain `ValueError` naming the field, the 1-based line number, and the offending line verbatim — mirroring `_raise_malformed`'s shape in `bcs.platform.adapters.efi.parser` exactly. A later duplicate line overwrites an earlier one (the same left-to-right overwrite semantics `FirmwareBootConfiguration`'s own scalar fields already have). `state=UNSUPPORTED` is never produced by this parser — that value is reserved for the not-yet-implemented adapter layer's Error Mapping (a non-zero exit/`stderr` condition, never stdout text). No cross-field validation, matching the model's own lack of a uniqueness constraint. Imports only `re`, `typing.NoReturn`, and its own `secureboot.models` — no `subprocess`, `CommandRunner`, `bcs.platform.adapters.efi`/`storage`, `bcs.inventory`, or `click`, verified by an AST-based purity test. 25 new tests (`cli/tests/test_platform_adapters_secureboot_parser.py`) against a `tmp_path`-rooted synthetic corpus loaded via `fixture_utils.py` (the real `cli/tests/fixtures/secureboot/` corpus stays untouched, still holding only its placeholder README), covering every recognized pattern, whitespace/CRLF tolerance, blank lines, unrecognized-line permissiveness, empty input, duplicate-line overwrite, both malformed-field rejections, and the import-purity check; `secureboot/parser.py` is at 100% statement and branch coverage. No `adapter.py`, `errors.py`, `__init__.py` re-export change, `RuntimeContext` integration, Host Discovery integration, CLI/inventory wiring, Boot Manager, Builder, or Deploy changes. Ruff, mypy (strict), and pytest all pass; full suite (697 tests) green.
